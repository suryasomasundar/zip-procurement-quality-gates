QA Delivery Process Overview

Zip QA Take-Home Summary
Repository: https://github.com/suryasomasundar/zip-procurement-quality-gates
Live application: https://zip-procurement-quality-gates.onrender.com

1. Purpose And Scope

This exercise is intentionally small at the product layer and more deliberate at the delivery layer. The goal is to demonstrate how a procurement workflow can be implemented with clear review standards, automated validation, protected merges, deployment confidence, and a lightweight release process.

The operating goal is simple: keep master releasable, make pull requests reviewable, and ensure releases are validated before they are marked complete.

2. End-To-End Process

Stage What Happens Owner
1. Feature branch An engineer creates a branch from master using an approved naming format such as feature/<change>. Author
2. PR creation The author opens a draft or ready-for-review PR with a ticketed title, completed template, validation notes, and risk summary. Author
3. Governance and CI GitHub Actions runs PR governance, CI quality checks, and security scans. Automation
4. Review Reviewers assess correctness, regression risk, test coverage, and deployment or workflow impact. Reviewers
5. Merge gate The PR may merge only after required approval, passing required checks, and resolved conversations. Author + reviewer
6. Deploy Merge to master triggers the same core validation and then Render deploys the application. Automation
7. Live validation The live URL, /api/health, and deployed smoke tests validate the running service. Automation + QA
8. Release A manual release workflow validates production and creates a version tag and GitHub Release. Release owner

3. Review Process And Working Agreement

Ready For Review Standard

Check Expectation
Branch naming Must follow approved pattern such as feature/<change>, fix/<change>, or docs/<change>.
PR title Must follow a ticketed format such as QA-1234: concise summary.
PR body Must include change summary, validation, checklist, and risk review sections.
Validation The author should run the relevant local checks and provide clear validation notes.
Review routing The author should request reviewers intentionally based on the affected files and change type.
Artifacts UI changes should include screenshots or other evidence where useful.

Communication Model

  • GitHub is the system of record for comments, approvals, checks, and merge blockers.
  • A dedicated Slack channel such as #team-prs should be used as the review attention channel.
  • When a PR is ready, the author should post the PR link, risk level, requested reviewers, and expected merge timing.
  • Urgent, release-blocking, or production-impacting PRs should also be called out in the main engineering channel.

Review SLA

Scenario Expectation
Standard PR First review response within 4 business hours
Urgent or release-blocking PR Acknowledged within 1 business hour
Author response to blocking comments Same business day when practical
Review delay beyond SLA Escalate to lead or engineering manager

4. Controls And Protections

Control Current implementation
PR gating Pull request required before merge
Approval At least 1 approval required
Conversation resolution Required before merge
Status checks Quality Gate plus governance and security checks
Merge policy Squash and rebase enabled, merge commits disabled
History protection Linear history required, force pushes blocked, deletions restricted
Ownership CODEOWNERS plus documented reviewer routing
Release control Production validation before tag and GitHub Release creation

5. Ways The Process Can Fail

Category Typical failure modes
Governance Invalid branch name, missing PR sections, missing ticket reference, stale reviewer ownership
CI Lint failures, type drift, broken unit tests, failed API or UI integration tests, missing coverage
Security Dependency vulnerabilities, CodeQL findings, accidental secret commits
Deployment Render build/startup failures, frontend asset path issues, health endpoint mismatch
Release Invalid tag format, failed deployed smoke tests, version drift between code and live service
Process Required check names drift from branch protection, review SLA slips, ownership is unclear during incidents

6. Notifications And Where They Go

Signal Current channel Recommended next step
PR check failures GitHub PR and GitHub Actions Slack post in team PR channel
Review requests GitHub notifications Slack ready-for-review message
Security findings GitHub Security and workflow logs Slack escalation for high-severity findings
Broken master GitHub Actions Engineering Slack alert and merge pause
Release validation failure GitHub Actions Release-specific Slack escalation

7. Stakeholders As The Process Evolves

Stakeholder Why they matter
Engineers Implement changes, respond to CI failures, and review code
QA / quality engineering Own test strategy, quality gates, flaky-test policy, and release confidence
Engineering managers Own review expectations, merge policy, and SLA enforcement
Platform / infrastructure Own CI/CD plumbing, deployment reliability, secrets, and workflow correctness
Security Advise on vulnerabilities, secret exposure, and risky workflow changes
Product / business stakeholders Validate that procurement routing and approval behavior still reflect business expectations

8. AI-Assisted Review

AI-assisted review is included as an optional advisory signal, not as a merge authority. It can help surface missing tests, workflow drift, release-risk changes, and suspicious configuration changes, but human reviewers remain responsible for correctness, approval, and release judgment.

  • Use AI review as an early warning signal.
  • Confirm actionable findings with a human reviewer.
  • Fix or explicitly address confirmed findings in the PR conversation.

9. Future Enhancements

  • Add preview environments for PR validation.
  • Add Slack notifications for broken master and release failures.
  • Add automatic reviewer routing by changed file path.
  • Add flaky-test quarantine and reliability tracking.
  • Add contract testing for API payload and behavior changes.
  • Add stricter release approvals for infrastructure and security-sensitive changes.
